# Copywriting

## Voice and Tone

- SHOULD use active voice: "Install the CLI" not "The CLI will be installed".
- SHOULD address users in second person ("you") rather than first person ("we").
- SHOULD frame positively; guide users to solutions rather than just stating problems.

## Error Messages

- SHOULD make error messages educational and actionable, not just descriptive.
- MUST guide users toward the exit: explain how to fix the problem, not just what went wrong.
- Example: Instead of "Invalid API key" say "Your API key is incorrect or expired. Generate a new key in your account settings."

## Casing

- SHOULD use Title Case for headings and button labels (Chicago style).
- SHOULD use sentence case for body text, descriptions, and helper text.
- MUST be consistent within a product; mixed casing looks unpolished.

## Punctuation

- SHOULD use curly quotes (" ") rather than straight quotes (" ").
- SHOULD use the ellipsis character (…) rather than three periods (...).
- SHOULD prefer "&" over "and" in headings and labels where space is tight.
- SHOULD end menu options and in-progress states with ellipsis: "Rename…", "Saving…".

## Numbers and Units

- SHOULD use numerals for counts: "8 deployments" not "eight deployments".
- MUST separate numbers and units with a non-breaking space: "10 MB" not "10MB".
- SHOULD use `font-variant-numeric: tabular-nums` when numbers need to align in columns or comparisons.

## Placeholders

- SHOULD use consistent placeholder formats: `YOUR_API_TOKEN_HERE` for strings, `0123456789` for numbers.

## Labels and Actions

- SHOULD make button labels specific: "Save API Key" rather than "Continue" or "Submit".
- SHOULD avoid ambiguous labels that require context to understand.
- SHOULD use consistent nouns; minimize unique terminology for the same concept.
- SHOULD lead with verbs for actions: "Install", "Create", "Delete".
